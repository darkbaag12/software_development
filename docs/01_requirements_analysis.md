# 01. 요구사항 분석

## 1. 시스템 목적

본 시스템은 소규모 음식점에서 고객의 웨이팅 등록, 호출, 테이블 배정, 입장 처리, 취소 및 노쇼 처리를 효율적으로 관리하기 위한 시스템이다.

단순히 고객 목록을 저장하는 것이 아니라, 고객 상태와 테이블 상태를 함께 관리하여 직원이 현재 입장 가능한 고객과 사용 가능한 테이블을 쉽게 판단할 수 있도록 지원한다.

## 2. 시스템 범위

### 2.1 포함 범위

| 구분 | 내용 |
|---|---|
| 고객 관리 | 웨이팅 등록, 상태 확인 |
| 직원 기능 | 대기 목록 조회, 고객 호출, 입장 처리, 취소/노쇼 처리 |
| 테이블 관리 | 테이블 등록, 상태 변경, 사용 가능 테이블 확인 |
| 배정 기능 | 고객 인원수에 맞는 테이블 추천 및 배정 |

### 2.2 제외 범위

| 제외 항목 | 제외 이유 |
|---|---|
| 예약 기능 | 과제 규모 축소를 위해 제외 |
| 실제 SMS 발송 | 외부 API 연동이 필요하므로 제외 |
| 실제 로그인/회원가입 | 권한 구조를 단순화하기 위해 제외 |
| 결제 기능 | 웨이팅 관리 목적과 직접 관련이 낮음 |
| 매출/정산 기능 | 시스템 범위를 벗어남 |

## 3. 이해관계자

| 이해관계자 | 설명 | 주요 관심사 |
|---|---|---|
| 고객 | 웨이팅을 등록하는 사용자 | 빠른 등록, 대기 상태 확인 |
| 직원 | 고객을 호출하고 테이블에 배정하는 사용자 | 대기 목록 관리, 테이블 배정, 예외 상황 처리 |
| 관리자 | 테이블 정보를 관리하는 사용자 | 테이블 등록/수정, 매장 상태 관리 |
| 개발팀 | 시스템을 구현하는 팀 | 명확한 요구사항, 구현 가능성, 테스트 가능성 |

## 4. 사용자 스토리

| ID | 사용자 스토리 |
|---|---|
| US-01 | 고객은 매장 입장을 기다리기 위해 이름과 인원수를 입력하여 웨이팅을 등록하고 싶다. |
| US-02 | 직원은 대기 고객을 순서대로 확인하기 위해 현재 웨이팅 목록을 보고 싶다. |
| US-03 | 직원은 입장 가능한 고객을 구분하기 위해 대기 고객을 호출 상태로 변경하고 싶다. |
| US-04 | 직원은 고객 인원수에 맞는 테이블을 빠르게 찾기 위해 시스템의 테이블 추천을 받고 싶다. |
| US-05 | 직원은 호출 후 오지 않는 고객을 관리하기 위해 노쇼 상태로 변경하고 싶다. |
| US-06 | 관리자는 실제 매장 구조에 맞게 테이블 정보를 관리하고 싶다. |

## 5. 기능 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-01 | 고객은 이름, 연락처, 인원수, 요청사항을 입력하여 웨이팅을 등록할 수 있어야 한다. | Must |
| FR-02 | 시스템은 웨이팅 등록 시 대기 번호를 자동 부여해야 한다. | Must |
| FR-03 | 인원수는 1명 이상이어야 한다. | Must |
| FR-04 | 웨이팅 등록 직후 고객 상태는 `waiting`이어야 한다. | Must |
| FR-05 | 직원은 대기 고객 목록을 등록 순서대로 조회할 수 있어야 한다. | Must |
| FR-06 | 직원은 `waiting` 상태 고객을 `called` 상태로 변경할 수 있어야 한다. | Must |
| FR-07 | 직원은 고객 웨이팅을 취소 처리할 수 있어야 한다. | Must |
| FR-08 | 직원은 `called` 상태 고객을 `no-show` 상태로 변경할 수 있어야 한다. | Should |
| FR-09 | 관리자는 테이블 번호와 수용 인원을 등록할 수 있어야 한다. | Must |
| FR-10 | 시스템은 고객 인원수 이상을 수용할 수 있는 `available` 테이블을 추천해야 한다. | Must |
| FR-11 | 직원은 호출된 고객에게 테이블을 배정하고 입장 처리할 수 있어야 한다. | Must |
| FR-12 | 입장 처리 후 고객 상태는 `seated`, 테이블 상태는 `occupied`가 되어야 한다. | Must |
| FR-13 | 직원은 테이블 상태를 `available`, `occupied`, `cleaning` 중 하나로 변경할 수 있어야 한다. | Should |

## 6. 비기능 요구사항

| ID | 요구사항 | 품질 속성 |
|---|---|---|
| NFR-01 | 주요 화면은 고객 목록과 테이블 현황을 직관적으로 보여주어야 한다. | 사용성 |
| NFR-02 | 대기 목록과 테이블 목록은 1초 이내에 표시되어야 한다. | 성능 |
| NFR-03 | 잘못된 입력값에 대해 명확한 오류 메시지를 제공해야 한다. | 사용성 |
| NFR-04 | 고객 상태와 테이블 상태는 서로 모순되지 않아야 한다. | 신뢰성 |
| NFR-05 | 핵심 업무 규칙은 테스트 케이스로 확인 가능해야 한다. | 시험성 |
| NFR-06 | 기능별 코드가 분리되어 유지보수가 가능해야 한다. | 유지보수성 |

## 7. 요구사항 분석 결과

본 시스템의 핵심 요구는 다음과 같이 정리된다.

1. 고객 웨이팅 상태를 명확히 관리한다.
2. 테이블 상태를 명확히 관리한다.
3. 고객 입장 처리 시 고객 상태와 테이블 상태가 함께 변경되도록 한다.
4. 인원수와 테이블 수용 인원을 기준으로 단순하지만 일관된 배정 규칙을 적용한다.
